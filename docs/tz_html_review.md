# Ревʼю ТЗ (HTML-драфт) для проєкту **12 Months**

## Критичні зауваження

1. **Невалідна HTML-структура у змісті**
   - Усередині `<ul class="toc-list">` вставлені `<div class="toc-divider"></div>`.
   - За специфікацією HTML у `<ul>` мають бути тільки `<li>`.
   - **Що виправити:** замінити divider на `<li class="toc-divider" aria-hidden="true"></li>` або винести divider за межі `<ul>`.

2. **Суперечність у плані запуску і термінах**
   - У hero вказано `MVP 4–5 тижнів`, а в Roadmap фаза MVP — `Тижні 1–5`, далі загальний план 13 тижнів.
   - **Що виправити:** узгодити єдиний таймлайн (наприклад, MVP = 5 тижнів, або змінити бейдж на 5 тижнів).

3. **Суперечність щодо scheduler стеку**
   - В архітектурі вказано `APScheduler`, а в стеку Redis-блоці — `Celery beat`.
   - **Що виправити:** залишити одну технологію для MVP (APScheduler **або** Celery beat) і прибрати другу з документу.

4. **Суперечність у схемі БД (тип `tags`)**
   - Поле `tags` описано як `TEXT[] (JSONB)` — це два різні типи.
   - **Що виправити:** обрати один тип:
     - або `TEXT[]` (простий список),
     - або `JSONB` (гнучка структура тегів).

## Важливі технічні ризики

5. **Не зафіксована валідація Telegram `initData`**
   - Для TWA критично перевіряти підпис `initData`, інакше можливий підроблений користувач.
   - **Що виправити:** додати окремий пункт у ТЗ: HMAC-перевірка `initData` на backend для кожного запиту від TWA.

6. **Оплати: немає вимоги про верифікацію webhook/signature провайдера**
   - Є згадка LiqPay/Monobank, але немає вимоги перевірки підпису callback.
   - **Що виправити:** прописати обовʼязкову signature verification + idempotency key для повторних callback.

7. **Webhook-реєстрація при старті FastAPI без уточнення multi-worker режиму**
   - При кількох воркерах може бути дублювання дій під час startup.
   - **Що виправити:** описати безпечний startup-процес (single init job / lock / окремий deploy script).

8. **Схема БД без індексів для гарячих запитів**
   - Є таблиці, але не прописані індекси на `orders(status, created_at)`, `important_dates(user_id, date)` тощо.
   - **Що виправити:** додати мінімальний перелік індексів в ТЗ.

## UX/UI та продукт

9. **Низька доступність (accessibility)**
   - Немає `aria-label` для частини інтерактивних елементів/іконок.
   - Частина тексту має низький контраст (наприклад, дрібний світло-сірий на світлому фоні).
   - **Що виправити:** додати базові a11y-вимоги в ТЗ (контраст WCAG AA, клавіатурна навігація, aria-мітки).

10. **Немає fallback-поведінки для `prefers-reduced-motion`**
    - У документі багато анімацій, але не вказано режим зменшення анімацій.
    - **Що виправити:** додати media-query `@media (prefers-reduced-motion: reduce)`.

11. **Мовно-редакційні неточності**
    - `/history`: «Остання 5 замовлення» → правильно «Останні 5 замовлень».
    - **Що виправити:** пройтись редактурою тексту перед фінальним затвердженням ТЗ.

## Рекомендований план виправлень

1. Виправити валідність HTML структури (списки/семантика).
2. Узгодити суперечливі частини ТЗ: терміни, scheduler, тип `tags`.
3. Додати розділ security requirements (initData, payment signatures, idempotency).
4. Уточнити DB design: індекси, обмеження, enum-стратегія в міграціях.
5. Додати non-functional requirements: a11y, performance, observability, backup/retention.

## Висновок

Документ сильний по візуальній частині і продуктовому баченню, але до технічної реалізації бракує чітких обовʼязкових вимог для **безпеки**, **узгодженості стеку** та **структури даних**. Після внесення правок ТЗ буде значно надійнішим для старту розробки.
